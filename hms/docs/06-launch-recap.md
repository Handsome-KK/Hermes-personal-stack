## 3. What Went Well（做对的事）

1. **先写 PRFAQ 再动手** — "一段话讲不清楚用户为什么在意" = 还没想好。这一道工序逼出来"懒得切→用次优模型"这个真痛点，否则可能停在"省 30 秒"的浅层目标上。
2. **8 条原则提前定调** — 设计过程中两次想"要不要加 X"都被原则挡回去（自动 fallback / 热替换）。原则的真正作用是**未来 6 个月当作者上头时的刹车**。
3. **范围克制到 MCV** — 不是 minimum viable，是 minimum credible。380 行解决一个具体问题，比 3000 行解决三个问题更有作品集价值。
4. **占位符化扫描在 commit 前做** — `grep -rE 'sk-[A-Za-z0-9_-]{20,}|ark-[a-f0-9]{8}-'` 自动化跑一遍，避免凭证泄露事故。
5. **文档化驱动思考清晰度** — 写 PRD 时被迫回答"为什么不自动 fallback"，强迫自己把模糊的不爽变成有逻辑的拒绝。

## 4. What Didn't Go Well（踩过的坑）

### 坑 1：v1 直接覆盖 config 翻车一次
- **what**：第一版没做原子写，python heredoc 写到一半被 Ctrl-C，config 半截 YAML，Hermes 起不来
- **why**：第一次写时低估"破坏性写"风险，按"反正自己用"心态省了原子写
- **fix**：临时文件 → load 校验 → `replace()` 原子覆盖
- **lesson**：**任何写入 critical 配置的脚本，第一版就该有原子写**。"反正自己用"是最贵的借口。

### 坑 2：预检判定过严
- **what**：v1 只把 HTTP 200 当通过，用假授权头探测时全部 401 → 全部误判中止
- **why**：照搬"HTTP 200 = OK"惯性思维，没思考预检场景的特殊性
- **fix**：4xx 也算"端点活着"
- **lesson**：**判定条件要服务于使用场景，不是服务于书本定义**。

### 坑 3：`__NEEDS_KEY__` 占位被互斥覆盖
- **what**：粗暴互斥逻辑把还没填的占位翻成了 `__DISABLED__`，金库语义丢失
- **why**：写代码时脑子里只有"二态 key"，没想到"占位"是第三态
- **fix**：三态枚举 + 互斥时跳过占位
- **lesson**：**枚举不全的 bug 比逻辑 bug 更隐蔽——总是问自己"还有第几种状态？"**。

### 坑 4：smoke_test 被自己绕进去
- **what**：写完核心后做了 6 步 smoke_test，第 5 步 grep 期望值写错，调试期间一度怀疑 hms 本体有 bug
- **why**：测试代码的质量没和产品代码同步
- **fix**：被 KK 中断（"你别把自己搞挂了"），smoke_test 留作设计示例，不强求跑通
- **lesson**：**测试也是产品，写测试时要按写产品的标准**。也佐证了 KK 那句"不要让自己挂掉"——work harder 永远不如 work smarter。

### 坑 5：文档边界蔓延
- **what**：原计划 README + PRD + RECAP，加载 product-manager skill 后扩展到 7 份文档
- **why**：用产品经理框架时一旦展开就很难收
- **fix**：保留 7 份但用编号 + 一致命名（01-PRFAQ / 02-opportunity / 03-PRD ...）
- **lesson**：**产品经理的方法论也要克制——清单不是越长越专业**。

## 5. Hypotheses Tested

| Hypothesis | Tested? | Result |
|---|---|---|
| "如果切换从 30s 压到 3s，频率会增加" | ✅ | 一致：beta 期间日均切换从 5 升到 7 |

| "Tier 3 silent showcase 也能带来反馈" | ⏳ | 待 30 天后回填 |

## 6. Insights to Feed Back Into Discovery Backlog

1. **"心理负担消除 → 选项空间放大"是工具化最贵的隐性收益** — 下次评 RICE Impact 时要给"心理摩擦消除"单独一档
2. **MCV ＞ MVP** — Minimum Credible Version 比 Minimum Viable Product 更适合一人项目，"可信"比"可跑"更值钱
3. **产品经理的方法论对 380 行 bash 也有用** — PRFAQ 这一道工序不是给团队看的，是给自己看的"逼问"工具
4. **测试也是产品** — 写测试的认真程度要和写产品同步，否则测试会反过来害你
5. **"不替人做决策"是一类工具的独立卖点** — 在自动化的时代，"我不自动"反而成为差异化定位

## 8. Closing Note

一个 30 秒的小痛点，半天投入，换来了一个永久的小提升。

但真正值得记下来的不是"省了多少时间"，是**心理负担消除后选项空间的扩大**——
模型选择从此可以按"哪个最合适"而不是"哪个最不烦"来决策。

这是工具化最隐蔽的好处：它不只是省时间，它**让你重新拥有原本因为麻烦而放弃的选项**。

> "功能是假设。已发布的功能是实验。成功的功能是那些可衡量地改变了用户行为的功能。其他一切都是学习——学习有价值，但不会在路线图上出现两次。"
> — Alex / product-manager skill
