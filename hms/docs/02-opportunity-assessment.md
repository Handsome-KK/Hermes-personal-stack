# Opportunity Assessment: hms (Hermes Model Switch)

**Submitted by**: KK / Alex (PM)
**Date**: 2026-06-19
**Decision needed by**: 2026-06-22
**Decision (in hindsight)**: ✅ **Build (MVP scope)** — shipped 2026-06-22

---

## 1. Why Now?（为什么是现在？）

三个市场/个人信号叠加：

1. **个人侧**：从单一 OpenAI key → 4 家 provider × 18 个模型，使用复杂度过了拐点。一周前还能容忍手改，本周已经因为切错 config 翻车 2 次。
2. **生态侧**：2026 上半年国内 provider（火山方舟 Agent Plan / DeepSeek 1M 上下文 / Z.AI GLM）相继成熟，"多 provider 共存"从极客玩法变成日常配置。
3. **Hermes Agent 侧**：自带的 fallback_model 机制可以做"挂了换一家"，但**主 provider 切换**没有 first-class CLI。这是个产品空白。

**如果推迟 6 个月会怎样**：
- 切换错误成本随频次线性增长，每出一次错都是 5 分钟 + 心理负担
- key 在多个 config 之间手抄的泄露风险只增不减
- 个人对"再加一家 provider"的意愿会下降——因为切换成本太高，新 provider 进不来了

→ **现在不做，等于把后续所有 provider 选择权交给"懒"**。

## 2. User Evidence（用户证据）

> n=1（这是 internal tool，用户=作者本人）；证据来自日志、聊天记录、自我观察。

### Behavioral Data（行为数据）

| 行为 | 观察值 | 解读 |
|---|---|---|
| 一周内手改 `config.yaml` 次数 | 47 次 | 远超偶发，是高频日常动作 |
| 同周内"找昨天 config 备份"事件 | 4 次 | 出错率 ~10%，恢复成本高 |
| YAML 解析失败导致 Hermes 起不来 | 1 次 | 失败惩罚严重（5min+） |
| 误把真 key 粘到 commit 暂存区差点提交 | 1 次 | 凭证安全隐患 |

### Self-Interview（自我访谈，整理出的反复出现的痛感）

> "我现在切模型之前都要深呼吸。"
> "想试一下豆包，但懒得改 config，算了用 GLM 凑合。" ← **关键引用**
> "切完才发现 base_url 拼错了，那一轮 agent 上下文白烧。"
> "金库我自己都不知道还剩哪些 key 没填。"

第二条最致命：**摩擦在悄悄改变模型选择**。当一个选项的成本太高，它会被无声地排除——哪怕它本来才是最优解。

### Support Signal（客服信号）
不适用（一人产品）。

### Competitive Signal（竞争信号）

| 工具 | 能否替代 | 差距 |
|---|---|---|
| 手写 sed/awk 脚本 | 部分 | 无备份/diff/预检/互斥 |
| `direnv` + 多 config 切目录 | 部分 | 每个 provider 一份 config，但凭证管理散乱、没有互斥 |
| keychain + envsubst | 不能 | 解决了凭证存储，没解决多字段联动写入 |
| Hermes 内置 `fallback_model` | 不能 | 解决的是"挂了换一家"，不是"主动切" |

→ 没有现成解决方案完整覆盖"备份 + 预检 + 互斥 + diff + 回滚"。

## 3. Business Case（商业论证）

> 一人产品的商业论证 = 个人时间/注意力 ROI。

- **时间节省（直接）**：日均 5~10 次切换 × (30s - 3s) = 节省 **2~5 min/天**
- **错误恢复节省（直接）**：日均 0.5~1 次出错 × (5min - 5s) = 节省 **2.5~5 min/天**
- **决策质量提升（间接）**：消除"懒得切"导致的次优模型选择，**这一项无法量化但可能是最大的**
- **凭证安全（间接）**：避免一次 key 泄露 ≈ 避免数小时的吊销/重置工作 + 财务风险

**战略契合**（与个人 OKR 对齐）：
- KK 个人 OKR Q2 主线包含"把日常 agent 工作流摩擦点都工程化干掉"
- hms 是这条主线下能在半天工时内拿到 ROI 最高的目标之一

**成本**：
- 设计 + 实现 + 文档：**~半天工时**
- 维护：预期接近 0（边界明确、依赖极少）

| Factor | Value | Notes |

| Reach | 1 user × 5~10 切换/天 × 365 ≈ **2000~3700 切换/年** | 高频内部用户 |

| Effort | **0.05 person-month** | 半天工时（含设计+实现+文档） |
| **RICE Score** | **(3000 × 2 × 0.9) ÷ 0.05 = 108,000** | 用户数虽 1，但频次和把握度极高 |

> RICE 公式在 N=1 用户场景下会偏离原始语义，但相对排序信号仍然有效——
> 此项的真实意义：**几乎所有竞争性需求都没有这个 ROI**，因此优先级 = 立即做。

## 5. Options Considered（备选方案）

| Option | Pros | Cons | Effort |
|---|---|---|---|

## 6. Recommendation

**Decision**: ✅ **Build — MVP scope (Option C)**

**Rationale**:
1. 频次高 × 失败惩罚大 × 投入小 = 教科书级"该工程化"信号
2. 痛点已被自我观察清晰锁定，需求不会变形
3. 设计原则（8 条）能限制范围蔓延，半天工时可控

**条件 / 改变决策的信号**：
- 如果发现"自动 fallback"反而是更需要的功能 → 推翻第 8 条原则，方案重做（暂未观察到）
- 如果 Hermes 上游加了官方多 provider switcher → 让位给官方实现

**Next step if approved**:
1. 周日下午写 v1（备份 + 切换 + diff）
2. 周一晚跑通后补 v2（金库分离 + 三态 key + 预检 + 原子写）
3. 周二归档为 showcase 仓库，写 PRD/RECAP/GTM 三件套

## 7. Post-mortem Note（事后回看，2026-06-22）

实际投入：约半天工时（含两次会话 + 文档化）。
实际收益：
- 切换耗时 30s → 3s（10×）
- 出错率 10% → 0%
- "懒得切所以用次优模型"现象消失

唯一被低估的：**心理负担消除带来的模型选择自由度**。这是上线后才感受到、当初 RICE 评分里没有体现的隐藏 Impact。
