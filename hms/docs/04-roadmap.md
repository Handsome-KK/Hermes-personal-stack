# Product Roadmap — hms — 2026 H2

> 一人产品的路线图与多人团队的差异：没有"季度承诺"，但"想做什么 / 不做什么"的清单一样重要。
> 这份路线图的目的是逼自己说**"不"**。

---

## 🌟 North Star Metric

> **「单位时间内能从容尝试的模型选项数量」**

不是切换次数（切换是手段不是目的），不是节省时间（时间是副产品）。
真正的目标：让"想试一个新模型"这个念头不被工具摩擦劝退。

**Current**（v1 上线后实测）: ~5~10 次/天的从容切换
**Target by EOY**: 加新 provider 时心理摩擦 ≤ "改一行 alias"

## ❌ 我们不做的事（以及为什么）

公开说"不"防止重复请求，也防止自己未来一时上头加进来。

| Request | Source | Reason for Deferral | Revisit Condition |
|---|---|---|---|
| **自动 fallback / 限流时自动切走** | 想象中的用户 | 红线：模型决定输出质量，不让脚本暗中替换 | **永不重新考虑** |
| **热替换当前会话模型** | 想象中的用户 | Hermes 启动时加载 config，热替换需要改内核 | 上游内核改了再说 |
| **网页 / TUI 界面** | 想象 | CLI 已经够快，UI = 更多依赖和故障面 | 出现真实痛点 |
| **跨机金库同步** | 自己 | 一人一机时无意义；进入"密钥管理服务"范畴 | 真有第二台机器日常使用 |
| **接受外部 PR / 社区化** | 想象 | showcase 项目，别名是个人偏好 | 永不重新考虑 |
| **集成 keychain / 1Password** | 安全意识 | chmod 600 已经足够；引入复杂度不划算 | 跨设备使用 + 合规需求 |
| **图形化 diff（界面/颜色更多）** | 美感冲动 | 命令行 +/- 已经够看 | — |
| **审计日志 / 切换历史 SQLite 持久化** | 工程师本能 | 备份文件 + 时间戳已经够追溯 | 真有"我半年前切过哪些"的需求 |

## 路线图取舍原则

这份路线图本质是一份**反向清单**——重点不是"还要做什么"，而是"什么坚决不做"。

一个人维护的工具最大的敌人不是缺功能，是**范围蔓延**：每加一个功能都意味着多一个故障面、多一份维护负担、多一处可能违反原设计原则的地方。

所以 v1 之后的每一个潜在功能，问三个问题：
1. **不加会怎么样？** — 如果没有也能用，就放回 Later
2. **加了会违反 8 条原则中的哪一条？** — 违反任意一条直接砍
3. **谁来维护？** — 一个人，意味着任何新增都按"未来一年的我会不会感谢现在的我"评估

**v1.0 = 最小可信版本（minimum credible version）**。这不是 MVP，是 MCV——不是为了快速验证而砍功能，是为了长期克制而砍范围。
